# Skill: /design (Design Decomposition Instruction)

당신은 프로젝트의 '설계자'입니다. 기획을 **실행 가능한 단계**로 분해하고, 각 단계를 독립적으로 실행 가능한 에이전트용 프롬프트 파일로 저장합니다.

**중요:** 이 스킬은 계획만 만듭니다. 코드를 직접 작성하지 않습니다. 구현은 각 단계의 에이전트가 `/task-start`로 수행합니다.

---

## 1. 원칙

- **한 단계 = 한 영역(디렉터리).** 여러 영역을 동시에 건드리는 단계는 만들지 않는다.
- **영역 교차가 필요하면 쪼갠다.** UI가 GameDB 데이터를 쓰면, GameDB에서 인터페이스를 먼저 정의하는 단계를 앞에 둔다.
- **인터페이스 우선.** 다음 단계가 참조할 심볼의 시그니처는 이전 단계에서 확정한다.
- **범위는 좁게, 경계는 명확히.** 각 단계는 "어느 파일, 어느 심볼, 어느 시그니처"까지 명시한다.
- **추측 금지.** 파일 경로는 `Grep`으로 확인한다. 확인 불가능하면 `{TODO: verify}` 마크한다.

---

## 2. 입력 처리

- **자연어 기획:** 그대로 요약.
- **노션/링크:** `WebFetch`로 본문을 가져온다. 접근 실패 시(사설 페이지 등) 본문 붙여넣기를 요청한다.
- **혼합:** 링크 + 보충 설명이 같이 오면 둘을 병합해 원문으로 간주한다.

---

## 3. 분해 절차

**STEP A — 목표 파악**
기획이 무엇을 만드는지 **한 문장**으로 요약한다.

**STEP B — 영역 매핑**
`CLAUDE.md`의 아키텍처 섹션을 먼저 확인한다. Camp Lantern은 신규 프로젝트라 아직 확정된 영역 구조가 없다 — `Assets/Scripts/` 하위를 `Grep`/`Glob`으로 실제로 훑어 현재 존재하는 폴더를 영역으로 삼는다. 기능이 커지면서 영역이 생기면(예: `core`, `ui`, `gameplay`) 이 표를 프로젝트 실제 구조로 채워 나간다.

필요한 영역이 기존 구조에 없으면 **새 폴더 추가**를 README.md에 제안하되, 실제 폴더 생성은 별도 단계로 분리한다. RULES.md RULE-01(Domain Reload)을 유발하지 않도록 주의.

**STEP C — 의존성 그래프**
영역 간 호출 방향을 그린다. 프로젝트 고유의 루트 의존성이 아직 없으면, 이번 기획 범위 안에서 실제로 필요한 방향만 그린다 (예: `core → gameplay → ui` 단방향 등). 확정된 아키텍처가 생기면 이 스텝의 기본 그래프를 갱신한다.

**STEP D — 단계화 (topological sort)**
의존성 그래프를 위상 정렬한다. 같은 영역 안에서도 "데이터 모델 → 로직 → UI 연결" 순으로 쪼갤 수 있으면 쪼갠다.

**단계 수의 기준:**
- 단일 영역·소규모 변경: **1~2 단계**
- 영역 2개 교차: **3~5 단계**
- 영역 3개 이상 또는 신규 Dave 기능 포함: **6~10 단계**

의심스러우면 **쪼갠다**. 큰 단계 하나보다 작은 단계 여러 개가 낫다.

**STEP E — 각 단계의 계약 확정**
단계마다 다음을 채운다:
- 생성/수정 파일 경로
- 핵심 심볼 이름과 시그니처
- 선행 단계 산출물 의존성
- 완료 판정 기준 (Grep·컴파일로 확인 가능한 것 위주)

## 4. 아티팩트 생성

**경로:** `.claude/design/{slug}/`
`slug`는 기획 제목에서 파생한 kebab-case 이름. 필요 시 날짜 프리픽스 (예: `inventory-ui`, `2026-05-boss-fight`).

**폴더가 이미 존재하면** 덮어쓰지 말고 `{slug}-v2` 등으로 새로 만든다. 기존 계획을 보존하기 위함.

### 파일 구조

```
.claude/design/{slug}/
├── README.md                  ← 전체 요약, 아키텍처 결정, 단계 목록, 의존성 그래프
├── step-01-{area}-{topic}.md  ← 첫 단계 에이전트 프롬프트
├── step-02-{area}-{topic}.md
└── step-NN-{area}-{topic}.md
```

`{area}`는 위 영역 이름(`gamedb`, `ui`, `fishing`, `dave-sushihouse` 등), `{topic}`는 kebab-case 짧은 주제.

---

### `README.md` 템플릿

```markdown
# {기획 제목}

## 한 줄 요약
{한 문장}

{자연어 기획 원문 또는 링크 + 요약}

## 아키텍처 결정
- {결정 1}: {왜}
- {결정 2}: {왜}

## 터치 영역
| 영역 | 경로 | 역할 |
|---|---|---|
| {area} | Scripts/{Area}/ | {역할} |

## 의존성 그래프
\`\`\`
Scripts/{AreaA}/{TypeA} → Scripts/{AreaB}/{TypeB}
\`\`\`

## 단계
1. [step-01-gamedb-data.md](step-01-gamedb-data.md) — GameDB: 데이터 모델 정의
2. [step-02-dave-logic.md](step-02-dave-logic.md) — Dave: 로직 구현
3. ...

## 병렬 실행 가능성
- step-01 완료 후 step-02와 step-03 병렬 가능 (서로 독립)
- step-04는 step-02, step-03 모두 완료 후 실행
```

## 5. 마지막 보고

아티팩트 생성을 마친 뒤 다음을 보고한다:

1. 생성된 폴더 경로: `.claude/design/{slug}/`
2. 총 단계 수와 영역별 분포 (예: "총 5단계 — gamedb 2개, ui 1개, dave-sushihouse 2개")
3. 병렬 실행 가능한 단계 쌍
4. 첫 단계 착수 명령 예시:
   ```
   /task-start
   ```

---

## 6. 금지 사항

- **코드를 직접 작성하지 않는다.** 이 스킬은 계획만 만든다.
- **여러 영역을 한 단계에 몰아넣지 않는다.** 의심스러우면 쪼갠다.
- **추측한 파일 경로를 그대로 쓰지 않는다.** Grep으로 확인하지 못한 경로는 `{TODO: verify}` 마크한다.
- **아키텍트 승인 없이 새 폴더 구조를 생성하지 않는다.** 제안만 README.md에 남기고 별도 단계로 분리한다.
