# domain/

이 프로젝트의 **기획 의도, 시스템 간 관계, 역사적 맥락**을 기록한다.

코드만 봐서는 알 수 없는 "왜 이렇게 설계됐는가"와 "건드리면 뭐가 같이 깨지는가"를
남기는 공간.

> GDD 원본은 `gdd/VR_코지_소셜_영지_게임_기획서.docx` (.docx라 직접 Read 불가 — 시스템별 md 문서가
> distilled 버전이다). 시스템 문서는 `.claude/INDEX.md` Level 2에 keywords와 함께 등록되어 있다.

## 이 폴더에 들어가는 것

- **기획 의도**: 왜 이 시스템이 이런 형태로 설계됐는가 (버린 대안 포함)
- **시스템 간 관계**: A가 바뀌면 B도 영향받는다 / A는 B를 읽지만 B는 A를 모른다
- **비즈니스 규칙**: 재화 상한, 레벨 공식, 저장 조건, 해금 조건 등
- **데이터 흐름 / 이벤트 체인**: 한 액션이 어떤 순서로 어떤 시스템을 거치는가
- **역사적 맥락**: 왜 이런 레거시가 남아있는가
- **플랫폼 특이사항**: 플랫폼별로 왜 다르게 동작하는가
- **용어 정의**: 이 프로젝트에서만 쓰는 단어

## 이 폴더에 **들어가지 않는** 것

- Unity/C# 일반 규칙 → CLAUDE.md 또는 `.claude/rules/scripts.md`
- 경로 규칙, 수정 금지 영역 → `.claude/rules/`
- 싱글톤 구조, 폴더 맵, 패키지 버전 → `CLAUDE.md`
- 코드 전문 붙여넣기 (코드는 소스에, 여기엔 **원리**만)

## 파일 구조

시스템 또는 데이터 단위로 분리:
```
domain/
├── README.md                  ← 이 파일
├── gdd/                       ← 기획서 모음 (도착 시 추가)
│   └── images/                ← 기획서 참고 이미지 (필요 시 Read 툴로 열람)
├── {system-name}.md           ← 예: save-system.md, currency-system.md
└── ...
```

(전부 미리 만들 필요 없음. GDD와 실제 시스템이 생기는 대로 필요한 것만 추가)

## 각 파일 템플릿

```markdown
# {시스템 이름}

## 한 줄 요약
{한 문장}

## 핵심 타입 / 진입점
- `ClassName` — `Assets/Scripts/{경로}/File.cs`
- `OtherClass` — {역할 한 줄}

## 시스템 간 관계
{이 시스템이 무엇과 연결되는가. 단방향인가 양방향인가.
 이걸 바꾸면 뭐가 같이 깨지는가.}

## 기획 의도 / 역사적 맥락
{왜 이 형태로 설계했는가. 어떤 대안을 버렸는가.
 레거시가 있다면 왜 남아있는가.}

## 숨은 규칙 / 암묵지
{코드에 주석으로 안 적혀있지만 실제로는 지켜야 하는 규칙.}

## 수정 시 주의
{깨기 쉬운 지점. 같이 수정해야 하는 파일.}
```

## 갱신 원칙

- 새로운 시스템을 설계하거나 기존 시스템의 의도를 파악했을 때 **즉시 기록**.
- Claude Code가 작업 중에 "이건 기록해둘 가치가 있다" 싶은 설계 결정을 발견하면 제안.
- 기존 파일을 수정할 때는 기획 의도 섹션부터 확인 — 의도를 모르고 바꾸면 대부분 버그.
- 새 파일을 추가하면 [`.claude/INDEX.md`](../INDEX.md)의 Level 2 섹션에도 이 파일의 `keywords`를 등록한다.

## CLAUDE.md와의 경계

| 내용 | 위치 |
|------|------|
| "OOO는 싱글톤이다" (구조 사실) | CLAUDE.md |
| "왜 OOO와 XXX가 나뉘어 있는가" (설계 의도) | domain/{system}.md |
| "네임스페이스는 이렇게 쓴다" (컨벤션) | CLAUDE.md 또는 rules/scripts.md |
| "재화는 property로만 수정한다" (코드 규칙) | rules/scripts.md |
| "왜 재화 변경 시 이 방식을 쓰는가" (설계 근거) | domain/{system}.md |
